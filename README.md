# StockPro — ระบบจัดการคลังสินค้า (Inventory Management System)

StockPro เป็นระบบจัดการคลังสินค้าแบบ multi-tenant (หลายองค์กรใช้ระบบเดียวกันได้
โดยข้อมูลแยกจากกันสมบูรณ์) รองรับผู้ใช้งานสองระดับ คือ **ผู้ดูแลระบบ (Admin)**
และ **พนักงาน (User)** ผ่านหน้า Login เดียวกันแต่แยกสิทธิ์การใช้งานตามบทบาท

> เอกสารนี้เขียนให้อ่านเข้าใจได้แม้ไม่มีพื้นฐานเขียนโปรแกรม — ถ้าต้องการรายละเอียด
> เชิงเทคนิคสำหรับพัฒนาต่อ (กติกาของโค้ด, ข้อควรระวังของแต่ละไลบรารี) ดูที่
> [`skill.md`](./skill.md)

## เริ่มต้นจากอะไร

โปรเจคนี้เริ่มจากโจทย์ "อยากได้ระบบสต็อกสินค้าที่ใช้งานได้จริง" — ไม่ใช่แค่หน้าตา
สวยงามแต่ต้องมีระบบสิทธิ์ผู้ใช้งานที่ถูกต้อง (แยก Admin/User), มีฐานข้อมูลจริง
รองรับหลายองค์กรพร้อมกัน, และมี flow การทำงานที่ครบวงจร ตั้งแต่ผู้ดูแลจัดการสินค้า
ไปจนถึงพนักงานเบิก-คืนสินค้าและผู้ดูแลอนุมัติ โดยมีภาพต้นแบบ (`inv.png`) เป็น
แนวทางด้าน UI/UX — หน้า Login แบบแท็บ Admin/User, แดชบอร์ดสรุปภาพรวม,
และกราฟแสดงการเคลื่อนไหวสินค้า

## สร้างขึ้นมาอย่างไร

โปรเจคพัฒนาเป็นลำดับขั้น (milestone) แต่ละขั้นต่อยอดจากขั้นก่อนหน้าโดยไม่ทิ้ง
โครงสร้างเดิม:

1. **วางรากฐาน (M0)** — สร้างโปรเจค Next.js 16 (App Router) ด้วย strict
   TypeScript, ออกแบบ schema ฐานข้อมูลแบบ multi-tenant ด้วย Prisma 7 บน
   Supabase PostgreSQL, วางระบบยืนยันตัวตนด้วย Auth.js (Credentials — อีเมล/
   รหัสผ่านธรรมดา ไม่ใช้ Google/Microsoft), และวาง layout หลักของแอป
2. **ระบบสินค้า (M1)** — สร้างฟีเจอร์จัดการหมวดหมู่และสินค้าแบบ CRUD ครบวงจร
   (เพิ่ม/แก้ไข/ลบ พร้อม validation และป้องกันข้อมูลข้ามองค์กร)
3. **แยกสิทธิ์ Admin/User และระบบเบิก-คืนสินค้า (M1.5)** — ออกแบบ
   หน้า Login ใหม่ทั้งหมดให้ตรงกับต้นแบบ (แท็บ Admin Login / User Login),
   สร้างแดชบอร์ดแยกสองแบบตามบทบาท, เพิ่มโมเดลข้อมูล `Requisition` สำหรับ
   ระบบเบิก-คืนสินค้าที่มีสถานะไหลตามลำดับ (รออนุมัติ → อนุมัติแล้ว → รอรับคืน →
   คืนแล้ว) พร้อมตัด/คืนสต็อกอัตโนมัติผ่าน stock ledger, และเพิ่มหน้าจัดการ
   ผู้ใช้งานสำหรับผู้ดูแลระบบ
4. **ระหว่างทาง** — ทดสอบทุก flow จริงผ่านเบราว์เซอร์ (ไม่ใช่แค่ตรวจโค้ด) และ
   พบ-แก้บั๊กจริงสองจุดที่มาจากต้นตอเดียวกันคือการเชื่อมต่อฐานข้อมูล Supabase
   สะดุดเป็นครั้งคราว: (ก) transaction หน่วงเกินค่า timeout เริ่มต้นของ Prisma
   ทำให้การอนุมัติรายการเบิกล้มเหลวเป็นบางครั้ง (ข) หน้าเว็บค้างได้นานถึง 20-30
   วินาทีเพราะไม่มี timeout ฝั่ง client เลย — แก้โดยกำหนด timeout ที่เหมาะสมทั้งสอง
   จุด ให้ระบบ "ล้มเหลวเร็วและแจ้งเตือนชัดเจน" แทนที่จะค้างเงียบ ๆ

## ระบบนี้ทำงานอย่างไร (ภาพรวมสำหรับผู้ที่ไม่ถนัดเรื่องโค้ด)

เว็บแอปพลิเคชันสมัยใหม่ส่วนใหญ่ (รวมถึงระบบนี้) แบ่งการทำงานออกเป็น 3 ส่วนหลัก
ลองนึกภาพเป็นร้านค้า:

| ส่วน | เปรียบเหมือน | มีหน้าที่ |
|---|---|---|
| **Frontend** (หน้าบ้าน) | หน้าร้านที่ลูกค้าเดินเข้ามาเห็น | สิ่งที่ผู้ใช้เห็นบนจอและคลิกโต้ตอบ — ปุ่ม, ฟอร์ม, ตาราง, กราฟ |
| **Backend** (หลังบ้าน) | พนักงานในร้านที่รับออเดอร์ไปจัดการ | ตรวจสอบสิทธิ์, ตรวจสอบความถูกต้องของข้อมูล, คำนวณ, ตัดสินใจว่าจะบันทึกอะไรลงฐานข้อมูล |
| **Database** (ฐานข้อมูล) | คลังสินค้าและตู้เก็บเอกสารของร้าน | เก็บข้อมูลถาวร — รายชื่อสินค้า, ผู้ใช้งาน, ประวัติการเบิก-คืน |

**จุดที่ต่างจากระบบทั่วไป**: โปรเจคนี้ไม่ได้แยก "โปรแกรมฝั่งหน้าบ้าน" กับ
"โปรแกรมฝั่งหลังบ้าน" เป็นคนละโปรแกรม/คนละเซิร์ฟเวอร์แบบดั้งเดิม แต่ใช้ **Next.js**
ซึ่งรวมทั้งสองฝั่งไว้ในโปรเจคเดียว — โค้ดที่รันบนเบราว์เซอร์ของผู้ใช้ (frontend) และ
โค้ดที่รันบนเซิร์ฟเวอร์เพื่อคุยกับฐานข้อมูล (backend) อยู่ในภาษาเดียวกัน
(TypeScript/React) และอยู่ในโปรเจคเดียวกัน ทำให้พัฒนาและดูแลง่ายกว่าการแยกเป็น
สองโปรเจค

## เทคโนโลยีที่ใช้ทั้งหมด (รายละเอียด)

### ฝั่ง Frontend (สิ่งที่ผู้ใช้เห็นและโต้ตอบ)

| เทคโนโลยี | ใช้ทำอะไร | เพราะอะไรถึงเลือกใช้ |
|---|---|---|
| **Next.js 16** (App Router) | เฟรมเวิร์กหลักของทั้งเว็บ — จัดการ routing (URL แต่ละหน้า), render หน้าเว็บ, และเป็นสะพานเชื่อมกับ backend ในตัวเดียวกัน | เฟรมเวิร์กมาตรฐานอุตสาหกรรมสำหรับ React ปัจจุบัน มีระบบ Server Components ที่ทำให้ดึงข้อมูลจากฐานข้อมูลได้โดยตรงจากฝั่งเซิร์ฟเวอร์อย่างปลอดภัย |
| **React 19** | ไลบรารีสร้าง UI แบบ component (แบ่งหน้าจอเป็นชิ้นเล็ก ๆ นำกลับมาใช้ซ้ำได้) | Next.js สร้างอยู่บนพื้นฐาน React |
| **TypeScript** (strict mode) | ภาษาเขียนโค้ดที่ตรวจสอบชนิดข้อมูลตั้งแต่ก่อนรัน (เช่น ป้องกันการส่งข้อความไปในช่องที่ต้องการตัวเลข) | ลดบั๊กที่เกิดจากพิมพ์ผิด/ส่งข้อมูลผิดชนิด ก่อนที่โค้ดจะไปถึงผู้ใช้จริง |
| **Tailwind CSS v4** | ระบบจัดการสไตล์/หน้าตา (สี, ระยะห่าง, ขนาด) ผ่านการเขียน class สำเร็จรูป | จัดหน้าเร็ว สม่ำเสมอทั้งเว็บ ปรับ dark mode ได้ในตัว |
| **shadcn/ui** | ชุดส่วนประกอบ UI สำเร็จรูป (ปุ่ม, ฟอร์ม, ตาราง, dialog, dropdown ฯลฯ) ที่เอาโค้ดมาไว้ในโปรเจคเราเอง แก้ไขเองได้เต็มที่ | ได้ UI ที่ดูเป็นมืออาชีพเร็ว โดยไม่ต้องเขียนทุกอย่างจากศูนย์ และไม่ผูกติดกับไลบรารีภายนอกที่แก้ไขไม่ได้ |
| **Radix UI** | กลไกเบื้องหลังของ shadcn/ui (การเปิด-ปิด dialog, การเข้าถึงด้วยคีย์บอร์ด, accessibility) | ทำให้ dialog/dropdown/select ใช้งานได้ถูกต้องตามมาตรฐาน accessibility โดยไม่ต้องเขียนเอง |
| **lucide-react** | ชุดไอคอนทางการ (ไม่ใช้อิโมจิในระบบ) | ไอคอนที่ดูเป็นมืออาชีพ สอดคล้องกันทั้งแอป ปรับสี/ขนาดได้อิสระ |
| **React Hook Form** | จัดการฟอร์ม (การกรอกข้อมูล, การตรวจสอบ, แสดง error) | ประสิทธิภาพดี จัดการฟอร์มซับซ้อน (เช่น ฟอร์มเบิกสินค้าหลายรายการ) ได้ง่าย |
| **Zod** | กำหนดกฎการตรวจสอบข้อมูล (เช่น อีเมลต้องถูกรูปแบบ, รหัสผ่านต้องยาวพอ) ใช้ร่วมกันทั้งฝั่งหน้าเว็บและฝั่งเซิร์ฟเวอร์ | ตรวจกฎเดียวกันสองรอบ (กันผู้ใช้กรอกผิด + กันการส่งข้อมูลมาตรงๆ โดยข้ามหน้าเว็บ) |
| **TanStack Table** | สร้างตารางข้อมูล (สินค้า, รายการเบิก, ผู้ใช้งาน) พร้อมการค้นหา/แบ่งหน้า | จัดการตารางข้อมูลจำนวนมากได้อย่างมีประสิทธิภาพ |
| **Recharts** | วาดกราฟ (กราฟเส้นแสดงรับเข้า/เบิกออกในแดชบอร์ด) | ไลบรารีกราฟที่ใช้งานง่ายและปรับแต่งสีให้เข้ากับธีมได้ |
| **Noto Sans Thai** | ฟอนต์ภาษาไทยจาก Google Fonts | ฟอนต์ไทยที่อ่านง่าย โหลดผ่าน Next.js โดยอัตโนมัติ ไม่ต้องติดตั้งเอง |

### ฝั่ง Backend (ตรรกะเบื้องหลัง — ทำงานบนเซิร์ฟเวอร์ ไม่ใช่บนเครื่องผู้ใช้)

โปรเจคนี้**ไม่มีเซิร์ฟเวอร์ backend แยกต่างหาก** (ไม่มีโฟลเดอร์ `server/` หรือ
โปรเจค Express/NestJS แยกอีกตัว) — ตรรกะฝั่งเซิร์ฟเวอร์ทั้งหมดอยู่ในโปรเจค Next.js
เดียวกัน ผ่านกลไกสองแบบ:

| เทคโนโลยี/กลไก | ใช้ทำอะไร |
|---|---|
| **Next.js Server Components** | โค้ดที่ดึงข้อมูลจากฐานข้อมูลมาแสดงผล — รันบนเซิร์ฟเวอร์เท่านั้น ผู้ใช้ไม่เห็นโค้ดส่วนนี้เลย (เช่น หน้ารายการสินค้าที่ดึงข้อมูลสต็อกจริงมาแสดง) |
| **Next.js Server Actions** | ฟังก์ชันที่รับคำสั่งจากผู้ใช้แล้วไปเปลี่ยนแปลงข้อมูลในฐานข้อมูล (เช่น "อนุมัติรายการเบิก", "เพิ่มสินค้าใหม่") — เขียนเรียกจากปุ่มในหน้าเว็บได้ตรง ๆ โดยไม่ต้องสร้าง API endpoint แยก |
| **Auth.js v5** | ระบบยืนยันตัวตน (login/logout) — ตรวจสอบอีเมล/รหัสผ่าน, ออก session (บัตรผ่านชั่วคราว) ให้ผู้ใช้ที่ login สำเร็จ, เก็บ session นั้นไว้ใน cookie ที่ปลอดภัย |
| **bcryptjs** | เข้ารหัสรหัสผ่านก่อนบันทึกลงฐานข้อมูล (ไม่มีการเก็บรหัสผ่านจริงไว้ที่ไหนเลย แม้แต่ในฐานข้อมูล) | 
| **Prisma 7** | ตัวกลางระหว่างโค้ด TypeScript กับฐานข้อมูล PostgreSQL — แปลงคำสั่งอย่าง "หาสินค้าที่คงเหลือน้อยกว่า 10 ชิ้น" ให้เป็นภาษา SQL ที่ฐานข้อมูลเข้าใจ พร้อมตรวจสอบชนิดข้อมูลให้ตรงกับ schema |

### ฐานข้อมูล (Database)

| เทคโนโลยี | ใช้ทำอะไร |
|---|---|
| **PostgreSQL** | ระบบฐานข้อมูลเชิงสัมพันธ์ (relational database) — เก็บข้อมูลเป็นตาราง มีความสัมพันธ์เชื่อมโยงกัน (เช่น สินค้าหนึ่งชิ้นอยู่ในหมวดหมู่ไหน, คำขอเบิกหนึ่งใบมีสินค้าอะไรบ้าง) เป็นฐานข้อมูลที่แม่นยำสูง เหมาะกับข้อมูลธุรกิจที่ต้องถูกต้อง 100% เช่น จำนวนสต็อก |
| **Supabase** | ผู้ให้บริการ PostgreSQL แบบพร้อมใช้บน cloud (ไม่ต้องติดตั้งฐานข้อมูลเองบนเครื่อง) มี dashboard ให้ดูข้อมูล จัดการ connection และมี "connection pooler" ที่ช่วยให้เว็บแอปเชื่อมต่อฐานข้อมูลได้พร้อมกันหลาย ๆ คำขอโดยไม่ล้น |
| **@prisma/adapter-pg** | ตัวเชื่อม (adapter) ที่ทำให้ Prisma คุยกับ Supabase ผ่านโปรโตคอลของ PostgreSQL โดยตรง |

## ฐานข้อมูลเก็บข้อมูลอะไรบ้าง

ระบบมีตารางข้อมูลทั้งหมด 11 ตาราง แบ่งเป็นกลุ่มตามหน้าที่:

**กลุ่มผู้ใช้งานและองค์กร**
| ตาราง | เก็บอะไร |
|---|---|
| `organizations` (องค์กร) | ชื่อองค์กร/ร้านค้า — ใช้แยกข้อมูลของแต่ละบริษัทออกจากกันโดยสมบูรณ์ |
| `users` (ผู้ใช้งาน) | ชื่อ, อีเมล, รหัสผ่าน (เก็บแบบเข้ารหัสเท่านั้น ไม่ใช่ข้อความจริง) |
| `memberships` (สมาชิกภาพ) | เชื่อมผู้ใช้แต่ละคนเข้ากับองค์กร พร้อมระบุสิทธิ์ (เจ้าของ/ผู้ดูแล/พนักงาน) |

**กลุ่มสินค้าและสต็อก**
| ตาราง | เก็บอะไร |
|---|---|
| `categories` (หมวดหมู่สินค้า) | ชื่อหมวดหมู่และคำอธิบาย เช่น "เครื่องเขียน", "อุปกรณ์ไอที" |
| `products` (สินค้า) | รหัสสินค้า (SKU), ชื่อ, หน่วยนับ, จำนวนคงเหลือ, จำนวนขั้นต่ำก่อนแจ้งเตือน, ราคาขาย, ราคาทุน |
| `stock_movements` (ประวัติการเคลื่อนไหวสินค้า) | บันทึกทุกครั้งที่สินค้าเข้า/ออก/ถูกปรับปรุงจำนวน พร้อมระบุว่าใครเป็นคนทำและเมื่อไร — เป็น "สมุดบัญชี" ที่ไม่มีการแก้ไขย้อนหลัง ใช้คำนวณยอดคงเหลือปัจจุบัน |

**กลุ่มเบิก-คืนสินค้า**
| ตาราง | เก็บอะไร |
|---|---|
| `requisitions` (คำขอเบิกสินค้า) | เลขที่คำขอ, สถานะ (รออนุมัติ/อนุมัติแล้ว/ไม่อนุมัติ/รอรับคืน/คืนแล้ว), ใครเป็นผู้ขอ, ใครเป็นผู้อนุมัติ |
| `requisition_items` (รายการสินค้าในคำขอ) | สินค้าและจำนวนที่ขอเบิกในคำขอแต่ละใบ (หนึ่งคำขอมีได้หลายสินค้า) |

**กลุ่มจัดซื้อ (ยังไม่เปิดใช้งานในหน้าจอ — เตรียมโครงสร้างไว้ล่วงหน้า)**
| ตาราง | เก็บอะไร |
|---|---|
| `suppliers` (ซัพพลายเออร์) | ชื่อ, อีเมล, เบอร์โทร, ที่อยู่ผู้จำหน่ายสินค้า |
| `purchase_orders` (ใบสั่งซื้อ) | เลขที่ใบสั่งซื้อ, สถานะ, ซัพพลายเออร์ที่สั่ง |
| `purchase_order_items` (รายการสินค้าที่สั่งซื้อ) | สินค้า, จำนวน, ราคาต้นทุนต่อหน่วยในใบสั่งซื้อแต่ละใบ |

โครงสร้างตารางทั้งหมดกำหนดไว้ในไฟล์เดียวคือ [`prisma/schema.prisma`](./prisma/schema.prisma)
— แก้ไฟล์นี้แล้วรัน `npm run db:migrate` ระบบจะสร้าง/ปรับตารางในฐานข้อมูลจริงให้อัตโนมัติ

## ฟีเจอร์ปัจจุบัน

**ผู้ดูแลระบบ (Admin / Owner)**
- แดชบอร์ดสรุปภาพรวม: จำนวนสินค้าทั้งหมด/คงเหลือ/ใกล้หมด/หมดสต็อก, กราฟ
  การเคลื่อนไหวสินค้า (รับเข้า vs เบิกออก) 30 วันล่าสุด, ตารางสินค้าใกล้หมด
- จัดการสินค้าและหมวดหมู่ (เพิ่ม/แก้ไข/ลบ)
- อนุมัติ/ปฏิเสธคำขอเบิกสินค้า และยืนยันการรับคืนสินค้า
- จัดการผู้ใช้งาน: เพิ่มบัญชีพนักงาน/ผู้ดูแล กำหนดสิทธิ์ ลบผู้ใช้งาน
- ตั้งค่าข้อมูลองค์กร

**พนักงาน (User)**
- แดชบอร์ดสรุปรายการเบิกของตนเอง พร้อมเมนูลัด
- ดู/ค้นหารายการสินค้าและสต็อกคงเหลือ (อ่านอย่างเดียว)
- เบิกสินค้า (ระบุสินค้าและจำนวนได้หลายรายการในคำขอเดียว)
- แจ้งคืนสินค้าที่เบิกไปแล้ว และติดตามสถานะคำขอของตนเอง

**อยู่ระหว่างพัฒนา** — ดูสถานะละเอียดใน [`skill.md`](./skill.md#roadmap--milestone-status):
รับเข้าสินค้า/รายการเคลื่อนไหวแบบเต็ม, รายงานและ export Excel, ซัพพลายเออร์และ
ใบสั่งซื้อ (หน้าเหล่านี้แสดงข้อความ "เร็ว ๆ นี้" ไว้ล่วงหน้า)

## โครงสร้างโปรเจค

```
src/app/(auth)/…        กลุ่มหน้า sign-in / sign-up
src/app/(app)/…         กลุ่มหน้าหลัก ต้องผ่านการยืนยันตัวตนก่อนเข้าถึง
src/features/<name>/    หนึ่งโฟลเดอร์ต่อหนึ่งฟีเจอร์ (schemas, queries, actions, components)
src/components/shared/  ส่วนประกอบใช้ร่วมกันทั้งแอป (DataTable, EmptyState, ConfirmDialog…)
src/components/ui/      shadcn/ui primitives ที่ generate มา
src/lib/                infrastructure ที่ใช้ร่วมกัน (Prisma client, auth, session, roles)
prisma/schema.prisma    โครงสร้างตารางฐานข้อมูลทั้งหมด
prisma/migrations/      ประวัติการเปลี่ยนแปลงโครงสร้างฐานข้อมูลทีละขั้น
```

หลักการสำคัญที่ต้องรักษาไว้เสมอ (แยก tenant ตาม organization, stock เป็น ledger
ไม่ใช่ตัวเลขที่แก้ตรง ๆ, ฯลฯ) อธิบายละเอียดใน [`skill.md`](./skill.md)

## วิธีติดตั้งและรัน (ละเอียด — ทำตามได้แม้ไม่เคยเขียนโปรแกรมมาก่อน)

### สิ่งที่ต้องเตรียมก่อน

1. **Node.js เวอร์ชัน 20 ขึ้นไป** — คือโปรแกรมที่ใช้รันโค้ด JavaScript/TypeScript
   บนเครื่อง ดาวน์โหลดได้ที่ [nodejs.org](https://nodejs.org) (เลือกเวอร์ชัน LTS)
   หลังติดตั้งแล้วตรวจสอบด้วยคำสั่งนี้ในหน้าต่าง Terminal/Command Prompt:

   ```bash
   node -v   # ต้องขึ้นเลขเวอร์ชัน เช่น v20.x.x หรือสูงกว่า
   ```

2. **บัญชี Supabase (ฟรี)** — คือบริการฐานข้อมูล PostgreSQL บนคลาวด์ที่โปรเจคนี้ใช้
   สมัครฟรีที่ [supabase.com](https://supabase.com) แล้วกด "New Project"
   (เลือก region ที่ใกล้ผู้ใช้งานจริงที่สุด เพื่อความเร็วในการเชื่อมต่อ)
   ตั้งรหัสผ่านฐานข้อมูลไว้ให้ดี (ต้องใช้ในขั้นตอนถัดไป)

3. **โปรแกรมแก้ไขโค้ด** (ไม่บังคับ แต่แนะนำ) — เช่น
   [Visual Studio Code](https://code.visualstudio.com) (ฟรี)

### ขั้นตอนติดตั้ง

**1. ติดตั้งไลบรารีทั้งหมดที่โปรเจคต้องใช้**

เปิด Terminal ในโฟลเดอร์โปรเจค แล้วรัน:

```bash
npm install
```

คำสั่งนี้จะอ่านรายชื่อไลบรารีทั้งหมดจากไฟล์ `package.json` แล้วดาวน์โหลดมาเก็บไว้ใน
โฟลเดอร์ `node_modules` (ใช้เวลาสักครู่ ขึ้นกับความเร็วอินเทอร์เน็ต)

**2. ตั้งค่าการเชื่อมต่อฐานข้อมูล**

คัดลอกไฟล์ตัวอย่างเป็นไฟล์ `.env` (ไฟล์นี้เก็บค่าลับ เช่น รหัสผ่านฐานข้อมูล
จะไม่ถูกอัปโหลดขึ้น git):

```bash
cp .env.example .env
```

เปิดไฟล์ `.env` ด้วยโปรแกรมแก้ไขข้อความ แล้วไปที่หน้า Supabase dashboard ของโปรเจค
ที่สร้างไว้ → เมนู **Connect** → แท็บ **ORMs** → เลือก **Prisma** จะเห็นค่า
connection string สองแบบให้คัดลอกมาใส่:

- `DATABASE_URL` — ใช้ตอนแอปทำงานจริง (แบบ **pooled**, พอร์ต `6543`, มี
  `?pgbouncer=true` ต่อท้าย) ช่วยให้รองรับผู้ใช้พร้อมกันหลายคนได้
- `DIRECT_URL` — ใช้ตอนสร้าง/แก้โครงสร้างตาราง (แบบ **direct**, พอร์ต `5432`)

จากนั้นสร้างค่า `AUTH_SECRET` (กุญแจลับสำหรับเข้ารหัส session ผู้ใช้) ด้วยคำสั่ง:

```bash
npx auth secret
```

คำสั่งนี้จะเติมค่าให้ในไฟล์ `.env` ให้อัตโนมัติ ไม่ต้องกรอกเอง

**3. สร้างตารางในฐานข้อมูล**

```bash
npm run db:migrate
```

คำสั่งนี้จะอ่านโครงสร้างตารางจาก `prisma/schema.prisma` แล้วไปสร้างตารางจริงบน
ฐานข้อมูล Supabase ที่ตั้งค่าไว้ในขั้นตอนที่ 2 — ถ้าสำเร็จจะเห็นข้อความว่า migration
ถูกสร้างและนำไปใช้แล้ว

**4. เริ่มรันระบบ**

```bash
npm run dev
```

รอจนขึ้นข้อความประมาณ `Local: http://localhost:3000` (ถ้าพอร์ต 3000 ถูกใช้งานอยู่
ระบบจะสลับไปพอร์ตอื่นให้อัตโนมัติ เช่น 3001) แล้วเปิดลิงก์นั้นในเบราว์เซอร์

**5. สมัครบัญชีแรกของคุณ**

ไปที่หน้า `/sign-up` (เช่น `http://localhost:3000/sign-up`) กรอกชื่อ, ชื่อองค์กร,
อีเมล, รหัสผ่าน — บัญชีนี้จะกลายเป็น **เจ้าของระบบ (Owner)** โดยอัตโนมัติ และเป็น
ผู้สร้างองค์กรใหม่ในระบบ

หลังจากนั้นเข้าเมนู **"จัดการผู้ใช้งาน"** เพื่อเพิ่มบัญชีพนักงานสำหรับทดสอบฝั่ง User
(เลือกสิทธิ์ "พนักงาน (User)" ตอนเพิ่ม) แล้วลอง login ผ่านแท็บ **User Login**
ในหน้า sign-in เพื่อดูมุมมองฝั่งพนักงาน

### แก้ปัญหาที่พบบ่อย

| อาการ | สาเหตุที่เป็นไปได้ | วิธีแก้ |
|---|---|---|
| `npm run dev` ขึ้น error เกี่ยวกับ database/connection | ค่าใน `.env` ผิด หรือยังไม่ได้รัน `npm run db:migrate` | ตรวจสอบ `DATABASE_URL`/`DIRECT_URL` ให้ตรงกับ Supabase dashboard |
| หน้าเว็บขึ้น "เกิดข้อผิดพลาด" เป็นบางครั้ง | การเชื่อมต่อ Supabase สะดุดชั่วคราว (ปกติสำหรับฐานข้อมูล cloud) | กดปุ่ม "ลองใหม่อีกครั้ง" — ถ้าเกิดถี่ผิดปกติ ตรวจสอบสถานะโปรเจคที่ Supabase dashboard |
| พอร์ต 3000 ถูกใช้งานอยู่ | มีโปรแกรมอื่นรันอยู่ที่พอร์ตนั้น (เช่นโปรเจคอื่น) | ไม่ต้องแก้อะไร Next.js จะสลับไปใช้พอร์ตถัดไปให้อัตโนมัติ (ดูเลขพอร์ตจากข้อความตอนรัน `npm run dev`) |

## คำสั่งที่ใช้บ่อย

```bash
npm run dev          # เริ่ม dev server (Turbopack) สำหรับพัฒนา/ทดสอบ
npm run build         # build โปรเจคสำหรับใช้งานจริง (production)
npm run start          # รันเวอร์ชันที่ build แล้ว (ต้อง build ก่อน)
npm run lint            # ตรวจสอบคุณภาพโค้ดด้วย ESLint
npm run typecheck        # ตรวจสอบความถูกต้องของชนิดข้อมูลด้วย TypeScript
npm run db:migrate       # สร้าง/รัน migration ใหม่ (ใช้ตอนแก้โครงสร้างฐานข้อมูล)
npm run db:studio        # เปิด Prisma Studio — หน้าเว็บสำหรับดู/แก้ข้อมูลในฐานข้อมูลโดยตรง
```

## พัฒนาต่อ

โปรเจคนี้จัดโค้ดแบบ "แยกตามฟีเจอร์" (`src/features/<ชื่อฟีเจอร์>`) — แต่ละฟีเจอร์มี
ไฟล์ครบชุดของตัวเอง (การตรวจสอบข้อมูล, การดึงข้อมูล, การบันทึกข้อมูล, หน้าตา UI)
ทำให้เพิ่มฟีเจอร์ใหม่หรือแก้ไขฟีเจอร์เดิมได้โดยไม่กระทบส่วนอื่น ก่อนเริ่มพัฒนาต่อ
แนะนำให้อ่าน [`skill.md`](./skill.md) ซึ่งรวบรวมกติกาสำคัญของโปรเจค
(เช่น ทุกการดึง/บันทึกข้อมูลต้องกรองตามองค์กรเสมอ, สต็อกสินค้าต้องเปลี่ยนผ่าน
ประวัติการเคลื่อนไหวเท่านั้น ห้ามแก้ตัวเลขตรง ๆ) และข้อควรระวังเฉพาะของไลบรารี
เวอร์ชันที่ใช้อยู่ในโปรเจคนี้
