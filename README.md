# StockPro — ระบบจัดการคลังสินค้า (เครื่องดื่ม)

StockPro เป็นระบบจัดการคลังสินค้าแบบ multi-tenant (หลายองค์กรใช้ระบบเดียวกันได้
โดยข้อมูลแยกจากกันสมบูรณ์) ออกแบบมาสำหรับคลังสินค้าประเภทเครื่องดื่มโดยเฉพาะ —
แต่ละสินค้าใส่รูปภาพประกอบได้เพื่อให้แยกแยะง่าย (เช่น ขวด/กระป๋อง/ลัง ที่หน้าตา
คล้ายกัน) หน่วยนับและตัวอย่างข้อมูลในฟอร์มก็ปรับให้เข้ากับสินค้าเครื่องดื่ม
(ขวด, กระป๋อง, ลัง) ระบบรองรับผู้ใช้งานสองระดับคือ **ผู้ดูแลระบบ (Admin)** และ
**พนักงาน (User)** ผ่านหน้า Login เดียวกันแต่แยกสิทธิ์ตามบทบาท

> เอกสารนี้เขียนให้เข้าใจง่าย ไม่ต้องมีพื้นฐานเขียนโปรแกรม — รายละเอียดเชิงเทคนิค
> สำหรับพัฒนาต่อ (กติกาของโค้ด, ข้อควรระวังของแต่ละไลบรารี) ดูที่
> [`skill.md`](./skill.md)

## สร้างขึ้นมาอย่างไร

โปรเจคพัฒนาเป็นลำดับขั้น ต่อยอดจากขั้นก่อนหน้าโดยไม่ทิ้งโครงสร้างเดิม:

1. **วางรากฐาน** — สร้างโปรเจค Next.js (App Router) ด้วย strict TypeScript,
   ออกแบบฐานข้อมูลแบบ multi-tenant ด้วย Prisma บน Supabase PostgreSQL, วางระบบ
   ยืนยันตัวตนด้วย Auth.js (อีเมล/รหัสผ่าน)
2. **ระบบสินค้า** — ฟีเจอร์จัดการหมวดหมู่และสินค้าแบบ CRUD ครบวงจร (เพิ่ม/แก้ไข/
   ลบ พร้อม validation และป้องกันข้อมูลข้ามองค์กร) รวมถึงรูปภาพสินค้า
3. **แยกสิทธิ์ Admin/User และระบบเบิก-คืนสินค้า** — หน้า Login แบบแท็บ, แดชบอร์ด
   แยกตามบทบาท, ระบบเบิก-คืนสินค้าที่มีสถานะไหลตามลำดับ (รออนุมัติ → อนุมัติแล้ว
   → รอรับคืน → คืนแล้ว) พร้อมตัด/คืนสต็อกอัตโนมัติผ่าน stock ledger
4. **ปรับแต่งให้เหมาะกับสินค้าเครื่องดื่ม** — เพิ่มฟิลด์รูปภาพสินค้า
   (ปรับขนาด/บีบอัดในเบราว์เซอร์ก่อนบันทึก ไม่ต้องใช้บริการเก็บไฟล์ภายนอก) และปรับ
   ตัวอย่างข้อมูลในฟอร์มให้ตรงกับสินค้าเครื่องดื่ม

ทุก flow ทดสอบจริงผ่านเบราว์เซอร์ (ไม่ใช่แค่ตรวจโค้ด) ก่อนถือว่าเสร็จ

## ระบบนี้ทำงานอย่างไร

เว็บแอปสมัยใหม่แบ่งการทำงานเป็น 3 ส่วน ลองนึกภาพเป็นร้านค้า — **แต่ระบบนี้ไม่มี
ลูกค้าเดินเข้ามาใช้งานเอง** เป็นเครื่องมือใช้ภายในร้าน คนที่นั่งหน้าจอคือ
**พนักงาน/ลูกน้อง** (จัดข้อมูลสินค้า, เบิก-คืนของ) และ **เจ้าของร้าน** (ดูภาพรวม,
อนุมัติ, จัดการพนักงาน) เท่านั้น:

| ส่วน | เปรียบเหมือน | มีหน้าที่ |
|---|---|---|
| **Frontend** (หน้าบ้าน) | หน้าจอ/เคาน์เตอร์ที่พนักงานและเจ้าของร้านนั่งทำงาน | ปุ่ม, ฟอร์ม, ตาราง, กราฟ — สิ่งที่ผู้ใช้คลิกโต้ตอบ |
| **Backend** (หลังบ้าน) | กฎและขั้นตอนที่ทำงานอยู่เบื้องหลังจอ | ตรวจสิทธิ์, ตรวจความถูกต้องของข้อมูล, ตัดสินใจว่าจะบันทึกอะไร — ทำงานเหมือนกันไม่ว่าใครเป็นคนกด |
| **Database** (ฐานข้อมูล) | คลังสินค้า/ตู้เก็บเอกสาร | เก็บข้อมูลถาวร — สินค้า, ผู้ใช้งาน, ประวัติเบิก-คืน |

**แยกให้ชัด**: ตารางข้างบนอธิบาย "ชั้นเทคโนโลยี" (โค้ดทำงานอย่างไร) ซึ่งเป็นคนละ
เรื่องกับ "สิทธิ์การใช้งาน" ของ Admin (เจ้าของร้าน) กับ User (พนักงาน) — ทั้งสอง
บทบาทใช้ Frontend หน้าเดียวกัน เพียงแต่ Backend จะควบคุมว่าใครเห็น/แก้ไขอะไรได้
บ้างตามสิทธิ์ของแต่ละคน (ดูรายละเอียดสิทธิ์ในหัวข้อ [ฟีเจอร์ปัจจุบัน](#ฟีเจอร์ปัจจุบัน))

โปรเจคนี้ใช้ **Next.js** ซึ่งรวมทั้งหน้าบ้านและหลังบ้านไว้ในโปรเจคเดียว — ไม่มี
เซิร์ฟเวอร์ backend แยกต่างหาก โค้ดทั้งสองฝั่งอยู่ในภาษาเดียวกัน
(TypeScript/React) ทำให้พัฒนาและดูแลง่ายกว่าการแยกเป็นสองโปรเจค

## เทคโนโลยีที่ใช้

### Frontend

| เทคโนโลยี | ใช้ทำอะไร |
|---|---|
| **Next.js 16** (App Router) | เฟรมเวิร์กหลัก — routing, render หน้าเว็บ, เชื่อมกับ backend ในตัวเดียว |
| **React 19 + TypeScript (strict)** | สร้าง UI แบบ component พร้อมตรวจชนิดข้อมูลตั้งแต่ก่อนรัน กันบั๊กจากพิมพ์ผิด/ส่งข้อมูลผิดชนิด |
| **Tailwind CSS v4** | จัดสไตล์/หน้าตาแบบ utility class รองรับ dark mode ในตัว |
| **shadcn/ui + Radix UI** | ส่วนประกอบ UI สำเร็จรูป (ปุ่ม, ฟอร์ม, ตาราง, dialog) ที่แก้โค้ดเองได้เต็มที่ พร้อม accessibility ที่ถูกต้อง |
| **lucide-react** | ชุดไอคอนทางการ ไม่ใช้อิโมจิในระบบ |
| **React Hook Form + Zod** | จัดการฟอร์มและกฎตรวจสอบข้อมูล ใช้กฎเดียวกันทั้งฝั่งเว็บและเซิร์ฟเวอร์ |
| **TanStack Table** | ตารางข้อมูล (สินค้า, รายการเบิก, ผู้ใช้งาน) พร้อมค้นหา/แบ่งหน้า |
| **Recharts** | กราฟแสดงการเคลื่อนไหวสินค้าในแดชบอร์ด |

### Backend

ไม่มีเซิร์ฟเวอร์แยก — ตรรกะฝั่งเซิร์ฟเวอร์ทั้งหมดอยู่ในโปรเจค Next.js เดียวกัน:

| เทคโนโลยี/กลไก | ใช้ทำอะไร |
|---|---|
| **Server Components** | ดึงข้อมูลจากฐานข้อมูลมาแสดงผล รันบนเซิร์ฟเวอร์เท่านั้น |
| **Server Actions** | ฟังก์ชันรับคำสั่งจากผู้ใช้แล้วเปลี่ยนแปลงข้อมูล (เช่น "เพิ่มสินค้า", "อนุมัติคำขอเบิก") เรียกจากปุ่มได้ตรง ๆ ไม่ต้องสร้าง API endpoint แยก |
| **Auth.js v5** | ยืนยันตัวตน (login/logout), ออก session ให้ผู้ใช้ที่ login สำเร็จ, เก็บใน cookie ที่ปลอดภัย พร้อมกำหนดอายุ session (9 ชั่วโมง) เพื่อความปลอดภัย |
| **bcryptjs** | เข้ารหัสรหัสผ่านก่อนบันทึก ไม่มีการเก็บรหัสผ่านจริงไว้ที่ไหนเลย |
| **Login rate limiting** | ล็อกอีเมลที่กรอกรหัสผ่านผิด 5 ครั้งติดกันชั่วคราว (15 นาที) ป้องกันการเดารหัสผ่านแบบอัตโนมัติ (brute-force) |
| **Prisma 7** | ตัวกลางระหว่างโค้ด TypeScript กับฐานข้อมูล PostgreSQL |

รูปภาพสินค้า/รูปโปรไฟล์ถูกปรับขนาดและบีบอัดในเบราว์เซอร์ก่อนส่ง แล้วเก็บเป็น
base64 โดยตรงในฐานข้อมูล — ไม่ต้องพึ่งบริการเก็บไฟล์ภายนอก (object storage)

### ฐานข้อมูล

| เทคโนโลยี | ใช้ทำอะไร |
|---|---|
| **PostgreSQL** | ฐานข้อมูลเชิงสัมพันธ์ เก็บข้อมูลเป็นตารางที่เชื่อมโยงกัน แม่นยำสูง เหมาะกับข้อมูลธุรกิจ เช่น จำนวนสต็อก |
| **Supabase** | ผู้ให้บริการ PostgreSQL บน cloud พร้อม connection pooler รองรับผู้ใช้พร้อมกันหลายคน |
| **@prisma/adapter-pg** | ตัวเชื่อม Prisma กับ Supabase ผ่านโปรโตคอล PostgreSQL โดยตรง |

โครงสร้างตารางทั้งหมดกำหนดไว้ที่ [`prisma/schema.prisma`](./prisma/schema.prisma)
ตารางหลัก ๆ ได้แก่ `organizations`, `users`/`memberships`, `categories`,
`products` (มีฟิลด์รูปภาพสินค้า), `stock_movements` (ประวัติเคลื่อนไหวสินค้าแบบ
ledger แก้ย้อนหลังไม่ได้), `requisitions`/`requisition_items` (เบิก-คืนสินค้า),
และ `suppliers`/`purchase_orders` (เตรียมโครงสร้างไว้ล่วงหน้า ยังไม่เปิดใช้หน้าจอ)

## ฟีเจอร์ปัจจุบัน

**ผู้ดูแลระบบ (Admin)**
- แดชบอร์ดสรุปภาพรวม: จำนวนสินค้า/ใกล้หมด/หมดสต็อก, กราฟการเคลื่อนไหวสินค้า
- จัดการสินค้าและหมวดหมู่ (เพิ่ม/แก้ไข/ลบ พร้อมแนบรูปภาพสินค้า)
- อนุมัติ/ปฏิเสธคำขอเบิกสินค้า และยืนยันการรับคืนสินค้า
- จัดการผู้ใช้งานและตั้งค่าองค์กร

**พนักงาน (User)**
- ดู/ค้นหารายการสินค้าและสต็อกคงเหลือ พร้อมรูปภาพช่วยแยกแยะสินค้า
- เบิกสินค้า (หลายรายการในคำขอเดียว) และแจ้งคืนสินค้าที่เบิกไปแล้ว
- ติดตามประวัติและสถานะคำขอของตนเอง

**อยู่ระหว่างพัฒนา** — รับเข้าสินค้าแบบเต็ม, รายงาน/export Excel, ซัพพลายเออร์และ
ใบสั่งซื้อ (ดูสถานะละเอียดใน [`skill.md`](./skill.md#roadmap--milestone-status))

## โครงสร้างโปรเจค

```
src/app/(auth)/…        กลุ่มหน้า sign-in / sign-up
src/app/(app)/…         กลุ่มหน้าหลัก ต้องผ่านการยืนยันตัวตนก่อนเข้าถึง
src/features/<name>/    หนึ่งโฟลเดอร์ต่อหนึ่งฟีเจอร์ (schemas, queries, actions, components)
src/components/shared/  ส่วนประกอบใช้ร่วมกันทั้งแอป (DataTable, EmptyState, ConfirmDialog…)
src/components/ui/      shadcn/ui primitives ที่ generate มา
src/lib/                infrastructure ที่ใช้ร่วมกัน (Prisma client, auth, session, roles)
prisma/schema.prisma    โครงสร้างตารางฐานข้อมูลทั้งหมด
prisma/migrations/      ประวัติการเปลี่ยนแปลงโครงสร้างฐานข้อมูลทีละขั้น
```

หลักการสำคัญ (แยก tenant ตาม organization, สต็อกเป็น ledger ห้ามแก้ตัวเลขตรง ๆ
ฯลฯ) อธิบายละเอียดใน [`skill.md`](./skill.md)

## วิธีติดตั้งและรัน

1. **ติดตั้ง Node.js 20 ขึ้นไป** จาก [nodejs.org](https://nodejs.org) แล้วตรวจสอบ
   ด้วย `node -v`
2. **สมัครบัญชี Supabase (ฟรี)** ที่ [supabase.com](https://supabase.com) →
   สร้างโปรเจคใหม่ → ตั้งรหัสผ่านฐานข้อมูลไว้ให้ดี
3. **ติดตั้งไลบรารี**

   ```bash
   npm install
   ```

4. **ตั้งค่าการเชื่อมต่อฐานข้อมูล** — คัดลอกไฟล์ตัวอย่างเป็น `.env`

   ```bash
   cp .env.example .env
   ```

   เปิดไฟล์ `.env` แล้วไปที่ Supabase dashboard → **Connect** → แท็บ **ORMs** →
   เลือก **Prisma** คัดลอกค่ามาใส่:
   - `DATABASE_URL` — แบบ pooled (พอร์ต `6543`) ใช้ตอนแอปทำงานจริง
   - `DIRECT_URL` — แบบ direct (พอร์ต `5432`) ใช้ตอนสร้าง/แก้โครงสร้างตาราง

   จากนั้นสร้างค่า `AUTH_SECRET` (จะเติมลงไฟล์ `.env` ให้อัตโนมัติ):

   ```bash
   npx auth secret
   ```

5. **สร้างตารางในฐานข้อมูล**

   ```bash
   npm run db:migrate
   ```

6. **เริ่มรันระบบ**

   ```bash
   npm run dev
   ```

   เปิด `http://localhost:3000` แล้วไปที่ `/sign-up` เพื่อสมัครบัญชีแรก
   (จะกลายเป็นเจ้าของระบบโดยอัตโนมัติ) จากนั้นเข้าเมนู **"จัดการผู้ใช้งาน"**
   เพื่อเพิ่มบัญชีพนักงานสำหรับทดสอบฝั่ง User

### แก้ปัญหาที่พบบ่อย

| อาการ | วิธีแก้ |
|---|---|
| `npm run dev` ขึ้น error เกี่ยวกับ database | ตรวจสอบ `DATABASE_URL`/`DIRECT_URL` ให้ตรงกับ Supabase dashboard และรัน `npm run db:migrate` |
| หน้าเว็บขึ้น "เกิดข้อผิดพลาด" เป็นบางครั้ง | การเชื่อมต่อ Supabase สะดุดชั่วคราว (ปกติสำหรับฐานข้อมูล cloud) — กด "ลองใหม่อีกครั้ง" |
| พอร์ต 3000 ถูกใช้งานอยู่ | ไม่ต้องแก้อะไร Next.js จะสลับไปพอร์ตถัดไปให้อัตโนมัติ |

## รันด้วย Docker

ใช้กรณีต้องการรันบนเครื่อง/เซิร์ฟเวอร์อื่นโดยไม่ต้องติดตั้ง Node.js เอง
(ต้องมี [Docker](https://www.docker.com) และตั้งค่า `.env` ตามขั้นตอนด้านบนไว้แล้ว
— `DATABASE_URL`, `DIRECT_URL`, `AUTH_SECRET`):

```bash
# รัน migration ครั้งแรก (หรือทุกครั้งที่ schema เปลี่ยน)
docker compose --profile tools run --rm migrate

# build + รันแอป
docker compose up -d --build
```

เปิด `http://localhost:3000` ได้ทันที — image ที่ build เป็นแบบ standalone
(Next.js เท่านั้น ไม่มีฐานข้อมูลอยู่ในนั้น เพราะโปรเจคนี้ต่อ Supabase บน cloud
อยู่แล้ว) `docker-compose.yml` อ่านค่าจาก `.env` ในโฟลเดอร์เดียวกันให้อัตโนมัติ

## คำสั่งที่ใช้บ่อย

```bash
npm run dev          # เริ่ม dev server (Turbopack) สำหรับพัฒนา/ทดสอบ
npm run build        # build โปรเจคสำหรับใช้งานจริง (production)
npm run start        # รันเวอร์ชันที่ build แล้ว
npm run lint         # ตรวจสอบคุณภาพโค้ดด้วย ESLint
npm run typecheck    # ตรวจสอบความถูกต้องของชนิดข้อมูลด้วย TypeScript
npm run db:migrate   # สร้าง/รัน migration ใหม่
```
